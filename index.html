LambdaCon subjects

Sessions will show how to leverage the power of functional programming languages such as Scala, Haskell, Clojure, Erlang, F#...
Nonetheless the conference aims to instill the functional mindset in attendees so that they can use it as a tool even when programming with an object oriented language

Attendees will learn basic and advanced concepts and how to use FP in their everyday life

  • immutability, higher order functions, currying, continuation-passing, pattern matching, monads
  • DSLs, domain modeling, actor model, machine learning
  • multithreading, non-blocking IO, message passing, message queues, microservices, HTTP-based APIs, web applications, high availability, continuous delivery
Conference structure

The program runs across multiple tracks
Sessions may be given in the form of a talk or a workshop

TALK
A presentation conducted in a lecture format and focused on a single topic: a language, technique or tool as well as an experience report
Primarly led by the speaker, a talk should also include some discussion

WORKSHOP
A hands-on session, focused around a specific language, technique, tool or issue, in which participants learn linking theory and practice by coding along with the speaker and experimenting

Learn more on LambdaCon

A conference for people eager to know more about functional programming, combining talks with hands-on sessions to take your skills to the next level

LambdaCon is focused on learning the functional approach, understanding in which scenarios it is better to use FP, discovering what can be done with current languages and technologies and exploring current best practices

Network with other funatics, learn something new, share your knowledge and experiences and have a lot of fun

Persisting persistent data structures

Reading and writing to data stores is often a pain point for programmers new to functional programming between the perceived loss of “purity” and the awkward mismatch between functional APIs and the relational data models or mutable documents in a document store.
Event sourcing is a well established alternative that meshes well with functional concepts of immutability and pure functions.
Using Event Store and F#, we’ll see how to use simple functional constructs like lists and folds to avoid the object-relational impedance mismatch and we’ll also investigate what makes for a "good" event to store, using techniques from Domain Driven Design.

The Actor Model in F# and Akka.NET

Writing correct concurrent, fault-tolerant, scalable applications is no simple task. The struggle is generally the result of using the wrong tools or wrong level of abstraction.
The Actor based concurrency model aims to simplify this process. The actor model is a mathematical model of concurrent computation originated in 1973 by Carl Hewitt, Peter Bishop, and Richard Steiger. Actors raise the abstraction level and provide a better platform to build correct concurrent and scalable applications.
This presentation will introduce concepts in architecture such as immutability, isolation and asynchronous processing utilizing tools such as Akka.NET which is a .NET port of the popular JVM framework Akka.

14:00 - 14:45
Generic polymorphism on steroids

Generic polymorphism allows to call the very same code on data structures that store data belonging to different types. Type inference allows to write code without declaring the type of functions: the compiler discovers by itself the most generic type of a polymorphic function.
New data types, however, still need to be declared beforehand. Can we push these mechanisms to let the compiler infer by itself the shape of data structures, completely removing the need for declaration of data types?
I will present how the OCaml language stretches the limits of type inference and generic polymorphism with polymorphic variant and the dual notion of extensible records. The former concisely solve, among others and with more guarantees, the expression problem often used to justify the supremacy of OOP. The latter provide persistent objects with user controlled late binding and a very flexible type system that is not based on subtyping and does not require dynamic casts.
